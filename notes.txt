On the Hierarchy: You're Right to Push Back

  The "specs immutable > tests read-only > code mutable" hierarchy
  solves a real problem (AI agents modifying test assertions to cheat)
   but it fundamentally misrepresents how TDD works.

  In Canon TDD, tests are not a fixed contract. They are discovery
  instruments. Beck's workflow is:

  1. Test list — behavioral analysis (what cases should this handle?)
  2. Write one test from the list
  3. Make it pass — and add new test needs to the list as you discover
   them
  4. Refactor
  5. Repeat until "fear transmutes into boredom"

  The test list is a living backlog, not a spec. Tests indicate next
  steps. You're writing, modifying, and learning from them
  continuously. Treating them as "read-only" kills the whole point.

  The real distinction is temporal, not categorical:
  - Tests you're currently writing in this TDD cycle → fully mutable
  (you're discovering)
  - Tests from previous work that already pass → protected from
  modification (the AI shouldn't "fix" these to accommodate new buggy
  code)

  That's the actual guardrail needed. Not "tests are read-only" but
  "don't modify existing passing tests to make new code work." The
  difference matters because one freezes the test suite, the other
  protects earned progress while allowing growth.

  On specs: I don't think you need to introduce "specs" as a new
  artifact. Your framework already has plan constraints and decisions
  — the things "decided during brainstorming that shouldn't be
  re-debated." That's your immutable layer. It's the intent that's
  immutable, not a formal spec document. If you call it a "spec"
  you'll invite the superpowers/spec-kit pattern of baking
  implementation details into it.

  ★ Insight ─────────────────────────────────────
  Beck's key nuance the repos miss: "When discovering new test needs
  during implementation, add them to the list. If a test invalidates
  prior work, decide whether to continue or restart with a different
  test order." This is tests as navigation, not tests as
  specification. The test list is a compass, not a map.

● ★ Insight ─────────────────────────────────────
  What makes this different from every repo we studied:
  1. No test code in plans — superpowers bakes exact test code into plans, violating both Canon TDD   
  and the planning skill's pointer principle
  2. Test list as compass, not map — none of the repos formalize the test list as a living behavioral 
  analysis that grows during implementation                                  
  3. Test ordering as a skill — none of the repos capture Beck's insight that sequence affects code
  structure
  4. "Fear → boredom" completion — every repo uses checklists; Canon TDD uses a psychological signal
  5. Lean (~160 lines) — superpowers uses ~600 lines mostly fighting rationalizations; enforcement
  moves to hooks, so the letskill just explains the method with WHY

  work.

tmux-worktree is an interesting implementation from ECC

  Your instinct is good. Let me refine:

  Where worktrees shine:
  - Parallel experiments: "Try approach A and approach B, compare results" — exactly your frontend
  intuition. Two different UI architectures, two different state management approaches. You compare, pick
   the winner, discard the loser.
  - Non-overlapping feature branches: auth + analytics + onboarding simultaneously. Each touches
  different files. Merges are clean.
  - Long-running test suites: Run the full test suite in one worktree while continuing development in
  another. No waiting.
  - Review + develop: Review a PR in one worktree while developing in another — no stashing/switching.

  Where worktrees don't help:
  - Sequential tasks within the same feature — shared state is what you want
  - Tightly coupled changes — if backend API changes require frontend changes, one worktree is better
  - Small tasks — overhead of branch + worktree + merge isn't worth it for 10-minute work


There is a tension between parallel execution and iterative adaptation is sharp. In
  parallel execution, one agent's realization can't propagate to siblings — they're isolated. This is the
   fundamental trade-off:

  - Sequential: Full knowledge propagation, slow
  - Parallel: Fast, but each agent operates on the original plan assumptions

  Your proposed solution (a "stop hook" when an agent encounters a problem needing user decision) is
  similar to what superpowers calls a "review checkpoint." The controller pauses all workers, surfaces
  the realization, user decides, agents continue with updated context.
