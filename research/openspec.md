## Structured Research Findings: Fission-AI/OpenSpec

### DIMENSION 1: Automation Outside .md Files

#### 1.1 The Architecture: CLI + Skills (Two-Layer System)

OpenSpec uses a **two-layer automation architecture**:

- **Layer 1: `openspec` CLI** (Node.js/TypeScript npm package) -- This is the scaffolding engine. It programmatically creates directories, generates skill/command files, manages schemas, tracks artifact state, and provides structured JSON output.
- **Layer 2: Generated SKILL.md / command .md files** -- These are AI instruction documents that the CLI writes into tool-specific directories (e.g., `.claude/skills/`). The AI reads these at invocation time and follows the instructions, which include calling the CLI for structured data.

The critical insight: **the .md files do NOT prescribe the directory structure directly. They instruct the AI to call CLI commands that scaffold the structure.**

#### 1.2 How Scaffolding Actually Works

When a user runs `openspec init`:

1. The CLI creates `openspec/` directory structure:
   ```
   openspec/
   ├── specs/
   ├── changes/
   │   └── archive/
   └── config.yaml
   ```

2. The CLI generates SKILL.md files into `.claude/skills/openspec-*/SKILL.md` (and equivalent for 20+ other tools like Cursor, Windsurf, Copilot, etc.)

3. The CLI generates command files via tool-specific adapters (e.g., `.claude/commands/` for Claude Code)

When the AI runs `/opsx:new add-dark-mode`, the skill instructs it to:
```bash
openspec new change "add-dark-mode"
```
This CLI command creates the change directory with `.openspec.yaml` metadata.

When the AI runs `/opsx:propose` or `/opsx:continue`, the skill instructs it to:
```bash
openspec status --change "add-dark-mode" --json
openspec instructions proposal --change "add-dark-mode" --json
```
The CLI returns structured JSON containing: template content, dependency info, project context, rules, output paths. The AI then writes the actual artifact files.

#### 1.3 Key Automation Mechanisms

| Mechanism | What It Does | File |
|-----------|-------------|------|
| `openspec init` | Scaffolds `openspec/` dir + generates skills/commands | `src/core/init.ts` |
| `openspec update` | Regenerates skills/commands when config/version changes | `src/core/update.ts` |
| `openspec new change` | Creates `openspec/changes/<name>/` + `.openspec.yaml` | `src/commands/change.ts` |
| `openspec status --json` | Returns artifact graph state (done/ready/blocked) | `src/core/artifact-graph/` |
| `openspec instructions --json` | Returns template + context + rules + deps for an artifact | `src/core/artifact-graph/instruction-loader.ts` |
| `openspec schemas` | Lists/manages workflow schemas | `src/commands/schema.ts` |
| `npm postinstall` | Auto-installs shell completions | `scripts/postinstall.js` |
| Command adapter registry | Generates tool-specific command files for 20+ AI tools | `src/core/command-generation/registry.ts` |

#### 1.4 The Skill-to-CLI Bridge Pattern

This is the most important pattern for our use case. Each SKILL.md is a **procedural instruction document** that tells the AI to:

1. Call CLI commands (bash) to get structured data
2. Parse JSON output
3. Read dependency files for context
4. Write artifact files following templates
5. Call CLI again to verify state

Example from `propose.ts` skill template:
```
1. openspec new change "<name>"          -- scaffolds directory
2. openspec status --change "<name>" --json  -- gets build order
3. For each artifact in dependency order:
   a. openspec instructions <id> --change "<name>" --json  -- gets template + rules
   b. Read dependency files
   c. Write artifact using template as structure
4. openspec status --change "<name>"     -- verify completion
```

The skill files are **generated by TypeScript code** in `src/core/templates/workflows/*.ts`, not hand-written. Each workflow has both a `SkillTemplate` (for Claude Code skills) and a `CommandTemplate` (for slash commands in other tools).

#### 1.5 Schema-Driven Configuration

Schemas live at `schemas/spec-driven/` and contain:
- `schema.yaml` -- Defines artifacts, their dependencies (DAG), output paths, templates, and instructions
- `templates/*.md` -- Skeleton files (just headings + HTML comments as placeholders)

The `schema.yaml` is machine-readable YAML, not consumed by the AI directly. The CLI's `ArtifactGraph` class (`src/core/artifact-graph/graph.ts`) does topological sorting (Kahn's algorithm) to determine build order. State detection is filesystem-based -- if the file exists, the artifact is "done."

#### 1.6 No Hooks Used

OpenSpec does **not** use Claude Code hooks (`settings.json`, `.claude/` hooks). The only non-.md automation is:
- npm `postinstall` hook (for shell completions)
- The CLI itself (invoked by the AI from within skill instructions)

### DIMENSION 2: Planning and Execution Patterns in .md Files

#### 2.1 The OpenSpec Format: Change Folders

Each "change" is a directory under `openspec/changes/<name>/` containing:
```
openspec/changes/add-dark-mode/
├── .openspec.yaml          # metadata (schema, created date)
├── proposal.md             # WHY: intent, scope, approach
├── specs/                  # WHAT: delta specifications
│   └── ui/
│       └── spec.md         # ADDED/MODIFIED/REMOVED requirements
├── design.md               # HOW: architecture decisions, trade-offs
└── tasks.md                # STEPS: implementation checklist with checkboxes
```

#### 2.2 Task Decomposition Pattern

The spec-driven schema defines a **DAG of 4 artifacts**:
```
proposal (root) ──> specs (requires proposal)
                ──> design (requires proposal)
                          └──> tasks (requires specs + design)
```

Each artifact has:
- **A template** (minimal skeleton with HTML comment placeholders)
- **An instruction** (detailed guidance in the schema.yaml's `instruction` field -- this is what the AI sees)
- **Dependencies** (which earlier artifacts must exist first)
- **A generates pattern** (output file path, e.g., `proposal.md`, `specs/**/*.md`)

#### 2.3 Multi-Session Execution

OpenSpec handles multi-session work through several mechanisms:

1. **Filesystem-based state**: Artifact completion is detected by file existence, not in-memory state. Any session can pick up where the last left off.

2. **Task checkboxes in tasks.md**: Progress tracking uses `- [ ]` / `- [x]` markdown checkboxes. The `apply` workflow parses these to determine what's done vs. pending.

3. **The `/opsx:apply` skill** instructs the AI to: read `tasks.md`, find unchecked items, implement them sequentially, check them off. If interrupted, the next session reads the same file and continues.

4. **Change folders persist across sessions**: All state lives in files, not in conversation context.

#### 2.4 Context Management Across Sessions

- **Project context**: Stored in `openspec/config.yaml` as a `context` field. Injected into every artifact instruction via the CLI's `--json` output.
- **Per-artifact rules**: Also in `config.yaml`, keyed by artifact ID. Only injected for matching artifacts.
- **Dependency chain**: Each skill instructs the AI to read completed dependency files before creating the next artifact. This rebuilds context from files rather than relying on conversation history.
- **The `openspec status --json` command**: Gives the AI a complete picture of what exists and what's blocked, no memory needed.

#### 2.5 Session Handoff Patterns

1. **Explicit handoff via task state**: `tasks.md` checkboxes serve as the handoff protocol. Any session reads the file and picks up unchecked items.
2. **Artifact state as handoff**: `openspec status --change --json` returns which artifacts are done/ready/blocked. A new session can orient itself in seconds.
3. **No explicit "session log"**: OpenSpec does NOT maintain a session log or handoff notes file. Context is reconstructed from the artifacts themselves.

#### 2.6 Delta Specs for Brownfield Work

A distinctive pattern: specs use ADDED/MODIFIED/REMOVED sections to describe changes relative to existing behavior. On archive, these deltas merge into `openspec/specs/`, which serves as the persistent "source of truth."

### KEY TAKEAWAY FOR OUR PLANNING SKILL

**OpenSpec's answer to "should the .md prescribe the structure or should a script scaffold it" is: BOTH, with clear separation of concerns.**

1. **A CLI/script scaffolds the directory structure and provides structured data** -- the AI never has to guess file paths or invent structure. The CLI does: `mkdir`, metadata creation, template loading, dependency graph computation, state detection.

2. **The SKILL.md is a procedural instruction that orchestrates CLI calls** -- it does NOT burn tokens describing the directory structure in prose. Instead it says "run this command, parse the JSON, write the file." The structure lives in code (TypeScript), not in the skill's markdown.

3. **Templates are minimal skeletons** -- just headings and HTML comment placeholders (50-200 bytes each). The AI fills them in. The _instructions_ for how to fill them live in the schema.yaml, delivered to the AI via CLI JSON output at invocation time, not baked into the skill.

4. **The token cost equation**: The SKILL.md files are substantial (the `propose` skill is ~4KB of instructions), but they contain procedural logic, not structure descriptions. The structure description cost is near-zero because it comes from CLI output only when needed.

The architectural recommendation from OpenSpec's design: **use a script/CLI to scaffold, and use the skill's .md to teach the AI how to call that scaffolder and populate the results.** This avoids both problems: the .md doesn't burn tokens on structure, and the user doesn't need to manually create directories. The OpenSpec CLI is ~30KB of TypeScript code dedicated to scaffolding, state detection, and template delivery.
